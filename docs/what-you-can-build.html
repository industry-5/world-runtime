What You Can Build with World Runtime

World Runtime is for teams building systems that need to understand the world they operate in, not just move tasks between screens.

Use it when you need explicit world state, append-only events, deterministic replay, simulation before action, and policy-governed decisions with evidence.

This page is vision-led. It separates repo-backed proof from adjacent opportunities the runtime is well-suited to support next.

Why this category exists

World-state systems solve a different class of problem

Ordinary workflow tools are good at routing tasks. World Runtime is built for systems where the state of the world itself matters: entities, relationships, events, projections, proposals, branches, and governed decisions.

Typical workflow software

Moves work between people and screens

  • state is often implicit inside forms, tickets, and service boundaries
  • history explains activity, but not always causality or consequence
  • approvals are often process steps, not hard runtime guardrails

World Runtime

Models the operating world, then governs action inside it

  • world state is explicit and rebuildable from append-only events
  • simulation branches compare futures before commitment
  • policy can deny, warn, or require approval before mutation

What you can build now

Three repo-backed proof paths already show the runtime under pressure

These are not hypothetical categories. They are the strongest public examples in this repo today, each exercising a different part of the runtime.

Repo-backed now

Supply network planning and disruption response

Model suppliers, routes, inventory, and delays as explicit world state. Rebuild history, compare reroute strategies, and keep what-if analysis separate from canonical state.

  • strongest proof: replay, projection, and simulation tradeoffs
  • constraint profile: medium policy, high branching and consequence tracing
  • good fit for operators who need to explain why outcomes changed

Safety-constrained air traffic coordination

Use policy-first controls where unsafe decisions must be denied or escalated and where evidence matters before action can be approved.

  • strongest proof: hard safety outcomes under time pressure
  • constraint profile: very high policy, explicit deny and require-approval paths
  • good fit for operations where unsafe branches cannot be allowed to slip through

Support and reality check

Ambition is broader than the support boundary

World Runtime has a clear public support posture. The stable runtime surfaces are not the same thing as the showcase story, and this distinction matters when you are evaluating production fit.

Stable and support-committed

  • App Server protocol
  • Public API /v1
  • Python SDK starter
  • Persistence migrations
  • Extension contracts

Experimental or best-effort

  • labs/ showcase surfaces and demos
  • storytelling assets under docs/
  • starter templates and exploratory examples unless promoted
  • release-candidate-only workflows that may still be refined

The World Game showcase stack is the primary polished demo story in this repo. It is valuable proof, but it is not the repo-wide support contract by itself.
